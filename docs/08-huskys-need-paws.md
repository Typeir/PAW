# Part 8: Huskys Need PAWs

> Git hooks should enforce the same quality gates for humans committing code as for agents editing files. PAW makes that automatic.

---

## The Husky Problem

Husky is the de facto standard for Git hooks in JS projects. It works — until it doesn't:

| Problem                      | What Happens                                                                                                                                |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| **Shell scripts everywhere** | `.husky/pre-commit` is a bash file. Windows developers hit CRLF issues, path quoting bugs, and missing utilities.                           |
| **Silent failures**          | `npx husky install` must run after `npm install`. If it doesn't, hooks simply don't exist. Nobody notices until broken code lands in main.  |
| **Git version fragility**    | Husky v9 switched to `core.hooksPath`. Some Git GUIs, older Git versions, and monorepo setups ignore it silently.                           |
| **No selective execution**   | Running all checks on every commit is slow. lint-staged helps but adds another config surface (`.lintstagedrc`) with its own failure modes. |
| **Agent blind spot**         | Husky enforces rules on `git commit`. Agents don't commit — they edit files via tools. So agents bypass every Husky check entirely.         |

That last point is the critical one. If quality enforcement only runs at commit time, and your workflow increasingly involves AI agents that never commit, your quality gates have a massive hole.

---

## PAW's Approach: One System, Two Surfaces

PAW already has a quality gate system (`pawGates.ts` + `.github/PAW/gates/`). Instead of maintaining a separate Husky + lint-staged pipeline, PAW provides a built-in Git hook configurator that calls the **exact same gates** agents run.

```
┌───────────────────────────────────────────────────────────┐
│                    .github/PAW/gates/                     │
│  file-length.gate.ts  antipatterns.gate.ts  test-gaps...  │
└──────────────────────────┬────────────────────────────────┘
                           │
              ┌────────────┼────────────────┐
              │            │                │
              ▼            ▼                ▼
        Agent hooks   Git hooks         CI pipeline
        (postToolUse,  (pre-commit,     (GitHub Actions)
         sessionEnd)    pre-push)
              │            │                │
              └────────────┼────────────────┘
                           │
                    pawGates.ts
                    same code, same gates,
                    same thresholds
```

### What This Means

- A developer running `git commit` gets the same checks an agent gets at session end
- A CI pipeline runs the same gates as both
- One set of `.gate.ts` files to maintain — zero drift between surfaces

## The PAW Git Hook Configurator

PAW includes a script that writes Git hooks directly — no Husky, no `core.hooksPath`, no `.husky/` directory. Plain Git hooks in `.git/hooks/`, written as thin shims that call `pawGates.ts`.

### Setup

```bash
# One command — writes pre-commit and pre-push hooks into .git/hooks/
paw hooks install
```

This is idempotent and can be wired into `package.json`:

```json
{
  "scripts": {
    "prepare": "paw hooks install"
  }
}
```

The `prepare` script runs automatically after `npm install` — same lifecycle Husky uses, but without the fragile `npx husky install` dance.

### What Gets Written

`pawHooks.ts install` writes executable scripts into `.git/hooks/`:

```bash
# .git/hooks/pre-commit (generated by PAW — do not edit)
#!/usr/bin/env bash
node .paw/hooks/paw-gates.mjs --changed-only --staged
```

```powershell
# .git/hooks/pre-commit (Windows — generated by PAW)
#!/usr/bin/env pwsh
node .paw/hooks/paw-gates.mjs --changed-only --staged
```

PAW detects the platform and writes the appropriate shim. The shim is intentionally minimal — all logic lives in the compiled `.mjs` gate runner, not in the hook script.

### The `--staged` Flag

When called with `--staged`, `pawGates.ts` scopes to files in the Git staging area (`git diff --cached --name-only`) instead of all uncommitted changes. This ensures only the code being committed is checked — not work-in-progress in the working tree.

```typescript
function getStagedFiles(rootDir: string): Set<string> {
  const output = execSync('git diff --cached --name-only --diff-filter=ACMR', {
    cwd: rootDir,
    encoding: 'utf-8',
    stdio: ['pipe', 'pipe', 'pipe'],
  }).trim();

  return new Set(output.split('\n').filter(Boolean));
}
```

This is integrated into `GateContext.targetFiles()` — when `--staged` is active, it intersects staged files with the gate's `appliesTo` extensions. No separate lint-staged config needed.

## pawHooks.ts — Implementation

Git hook configurations are loaded from `.paw/config.json` under the `gitHooks` key. This makes hook configuration project-specific without editing PAW framework code.

### Config Schema

```jsonc
// .paw/config.json
{
  "gitHooks": [
    {
      "name": "pre-commit",
      "flags": "--changed-only --staged",
      "description": "Run quality gates on staged files before commit",
      "appendCommands": ["node .paw/git-hooks/submodule-guard.mjs"],
    },
    {
      "name": "commit-msg",
      "description": "Validate commit message format",
      "command": "node .paw/git-hooks/commit-msg.mjs \"$1\"",
    },
    {
      "name": "pre-push",
      "flags": "--changed-only",
      "description": "Run quality gates and type-check before push",
      "appendCommands": ["npx tsc --noEmit"],
    },
  ],
}
```

### HookConfig Interface

```typescript
interface HookConfig {
  /** Git hook name (pre-commit, commit-msg, pre-push, etc.) */
  name: string;
  /** pawGates.ts flags for this hook (e.g. '--changed-only --staged') */
  flags?: string;
  /** Description shown during install */
  description: string;
  /** Override the entire hook command (replaces the default pawGates call) */
  command?: string;
  /** Additional commands appended after the main hook command */
  appendCommands?: string[];
}
```

When `command` is set, the hook runs that command instead of calling `pawGates.ts`. When `flags` is set (and no `command` override), the generated script calls `pawGates.ts` with those flags. `appendCommands` adds extra lines after the main command — useful for type-checking, LFS, or custom validators.

### Fallback Behaviour

When `.paw/config.json` has no `gitHooks` key or is absent, `pawHooks.ts` falls back to a single pre-commit hook with `--changed-only --staged`.

```typescript
function loadHookConfigs(): HookConfig[] {
  // 1. Read .paw/config.json
  // 2. If gitHooks array exists, return it
  // 3. Otherwise, return default pre-commit only
  return [
    {
      name: 'pre-commit',
      flags: '--changed-only --staged',
      description: 'Run gates on staged files',
    },
  ];
}
```

### Generated Scripts

```bash
# .git/hooks/pre-commit (generated by PAW — do not edit)
#!/usr/bin/env bash
# generated by PAW — do not edit
# Run quality gates on staged files before commit

node .paw/hooks/paw-gates.mjs --changed-only --staged
node .paw/git-hooks/submodule-guard.mjs
```

```bash
# .git/hooks/commit-msg (generated by PAW — do not edit)
#!/usr/bin/env bash
# generated by PAW — do not edit
# Validate commit message format

node .paw/git-hooks/commit-msg.mjs "$1"
```

### Safety: Non-PAW Hook Protection

If a project already has hand-written Git hooks, `pawHooks.ts` won't overwrite them. It checks for the `PAW_MARKER` comment — only hooks that PAW generated are managed by PAW. This means:

- Migrating from Husky? Remove `.husky/` and the `core.hooksPath` config, then run `pawHooks.ts install`
- Have custom hooks? PAW skips those and warns you
- Want to stop using PAW hooks? `pawHooks.ts uninstall` removes only what PAW wrote

## Configuring Which Gates Run

Not all gates make sense at commit time. The heavy gates (test-gaps, full content checks) should run at push or in CI, not on every commit. Configure this in `.paw/config.json` under `gitHooks`:

```jsonc
// .paw/config.json
{
  "gitHooks": [
    {
      "name": "pre-commit",
      "flags": "--gates file-length,antipatterns,jsdoc-quality --staged",
      "description": "Fast gates on staged files",
    },
    {
      "name": "pre-push",
      "flags": "--changed-only",
      "description": "All gates on changed files before push",
    },
  ],
}
```

This gives you a fast pre-commit (<2 seconds for regex-based gates) and a thorough pre-push.

### Suggested Gate Distribution

| Surface               | Gates                                    | Typical Time |
| --------------------- | ---------------------------------------- | ------------ |
| **pre-commit**        | file-length, antipatterns, jsdoc-quality | < 2s         |
| **pre-push**          | All gates, `--changed-only`              | 5–15s        |
| **Agent postToolUse** | Quick regex (inline, not full gates)     | < 1s         |
| **Agent sessionEnd**  | All gates, `--changed-only`              | 5–15s        |
| **CI pipeline**       | All gates, full codebase                 | 15–60s       |

The first three columns are fast enough that developers never notice. The last two are thorough enough that nothing slips through.

## Comparison: Husky vs PAW

| Concern             | Husky + lint-staged                           | PAW Git Hooks                                   |
| ------------------- | --------------------------------------------- | ----------------------------------------------- |
| Hook scripts        | Shell files in `.husky/`                      | Generated shims in `.git/hooks/`, one line each |
| Cross-platform      | Bash assumed, Windows issues common           | `node .mjs` command identical everywhere        |
| Setup reliability   | `npx husky install` must run post-install     | `prepare` script, same as Husky but simpler     |
| Config surfaces     | `.husky/`, `.lintstagedrc`, `package.json`    | `HOOK_CONFIGS` array in `pawHooks.ts`           |
| Staged file scoping | lint-staged                                   | `--staged` flag in `pawGates.ts`                |
| Agent enforcement   | None — agents don't commit                    | Same gates via Copilot hooks                    |
| Selective execution | Complex lint-staged glob config               | `--gates name,name` flag                        |
| Adding a check      | Edit lint-staged config + possibly Husky hook | Drop a `.gate.ts` file (same as always)         |
| Git version issues  | `core.hooksPath` can be ignored               | Plain `.git/hooks/` — works everywhere          |
| Uninstall           | Remove `.husky/`, unset `core.hooksPath`      | `pawHooks.ts uninstall`                         |

## Migration from Husky

```bash
# 1. Remove Husky
npm uninstall husky
rm -rf .husky

# 2. Remove lint-staged if no longer needed
npm uninstall lint-staged
# Delete .lintstagedrc.* or lint-staged config from package.json

# 3. Clean up core.hooksPath (if Husky v9 set it)
git config --unset core.hooksPath

# 4. Install PAW hooks
paw hooks install

# 5. Wire into prepare script
# In package.json:
#   "prepare": "paw hooks install"
```

## Skipping Hooks (Escape Hatch)

Sometimes you need to commit without gates — WIP commits, emergency hotfixes.

```bash
# Standard Git escape hatch — works with PAW hooks
git commit --no-verify -m "wip: checkpoint"
git push --no-verify
```

PAW doesn't fight this — `--no-verify` is Git's built-in mechanism and it works because PAW uses plain `.git/hooks/` with no tricks.

## Why Not Just Use Husky?

Husky is fine for simple setups. PAW replaces it when:

1. **You have AI agents** — They need the same gates humans get, and they don't run `git commit`
2. **You want one gate definition** — Not `.gate.ts` for agents AND `.lintstagedrc` for humans AND CI config for pipelines
3. **You hit Windows issues** — PAW's tsx approach eliminates an entire class of cross-platform bugs
4. **You want gate selection** — `--gates file-length,antipatterns` beats lint-staged's glob-based configuration
5. **You want reliability** — Plain `.git/hooks/` works with every Git client, every Git version, every GUI

If none of those apply, Husky works fine. PAW's position isn't "Husky is bad" — it's "if you already have PAW gates, why maintain a second system?"
